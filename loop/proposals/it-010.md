## 2. The printed evidence (measured on the page, and the diagnosis corrected)

seattle filled p1, baseline 299.33: `Employee Name:`[71.9–133.3]
`Marisol Vega`[138.3–185.0] `Job Title:`[311.7–344.1] `Prep Cook`[349.1–386.9]
— four runs, 8pt type. `EMPLOYEE NAME` is in `LABEL_SYNONYMS`; the value run is
sole in its cell (the unrecognised caption `Job Title:` closes it, the existing
barrier); the gap is 5.0pt, a word space, under `SIDE_BY_SIDE_MIN_GAP` (12.0).

**The backlog's mechanism note was a lead, not a fact — re-verified, the page
refuses the read TWICE, and the gap is the second refusal, not the first.** All
four runs are caption-shaped (`_is_caption_cell`: no digit, label-shaped), so
`_header_row_words` classifies the whole fill-in line as one of the page's own
column-header rows (≥3 caption cells). `Marisol Vega` thereby becomes a "header
cell": `_first_caption_run` makes it the cell's own barrier (leaving zero runs)
and `_caption_refusal` would refuse it as a value even at any gap. A licensed
rule must answer both refusals or flip nothing.

What distinguishes this line from a real column-header row, in print: the
colons. A column-header cell names the column BELOW it (`EMPLOYEE | PAY DATE |
REGULAR HOURS` — no colons, the corpus over); a colon-terminated caption is the
page punctuating "what follows me on my line is my value" — the same reading
`_caption_refusal` already gives the colon in the opposite direction (a
colon-run may never BE a value). Fill-in-the-line forms put values exactly
there; a filler writing at the blank after a colon produces exactly a
word-space gap.

## 3. The license (exactly one of the five): **printed label**

A **colon-terminated recognised label** may read the immediately following run
on its own baseline at a word-space gap, and for that read its own line is a
fill-in line, not a column-header row. Every other refusal stands, verbatim:
only the single run in the cell (two candidates tie → refuse); a colon-run
barrier still closes the cell (a colon-label followed by ANOTHER colon-label
reads nothing — the cell between them is empty); a colon-terminated candidate
is still never a value; header-row membership from ANY OTHER line still refuses
(the own-line amnesty reaches exactly the line the label itself punctuates);
the it-009 parallel-caption refusal still applies; and the run must pass the
field's parse gates. Bare labels (no colon) keep the 12.0pt column-gap
requirement and full header-row protection unchanged.

The prose hazard, held to measurement: prose that opens with a colon-carrying
phrase self-refuses in layers — a word-spaced sentence is ONE run, so the label
is not a separate run and matches no table key; a non-vocabulary caption
(`NOTE:`, `Job Title:`) is not a recognised label; and a licensed read must
still parse. Swept over all 77 + filled dev with the full extractor: the joint
rule changes ONE field on ONE document (the target). The instructional confirm
documents (osu/md_labor/lcc/kcha, colon-dense prose) move nothing.

## 4. The rule (one seam, inside `core/extract.py`)

Flag **`REALDOOR_COLON_GAP`** — on by default; `=0` restores bit-identical
behaviour. `_colon_fill_license(label_run)`: flag on AND the label run's
printed text ends with `:`.

- `_side_by_side_run`: the final gap test refuses only when the gap is under
  `SIDE_BY_SIDE_MIN_GAP` **and** the label is not colon-licensed. (Run
  separation already implies a printed gap ≥ the 0.6×size split threshold.)
- `_side_by_side_value`: when the label is colon-licensed, `header_words`
  minus the label's own line's words is used for BOTH the barrier inside
  `_side_by_side_run` and the `_caption_refusal` inside `_build_value_field` —
  the own-line amnesty. Other lines' header rows keep their full force, and the
  period-span path (`_period_span_fields`) keeps unreduced header words.
- The interior-colon refusal (found by this iteration's own hand-built-geometry
  test, P4): when the amnesty is engaged, a candidate carrying a
  colon-terminated TOKEN anywhere inside it is refused — the shape is the next
  caption merged with its own fill into one run (`Job Title: Prep Cook` at a
  word-space gap), which the amnesty would otherwise hand back after stripping
  its header-cell refusal. A licensed read is a clean fill or nothing.
  Refusal-biased: the committed conduct refused that run too.

No other function changes; `columns.column_value` (the visual-row reader) keeps
its 12.0pt gap — the colon licenses only the label's own line.

## 5. The hazards, named (each with the refusal that carries it)

- **Prose after a colon** (the named hazard: "PAY DATE: has not been assigned
  yet"): three independent refusals — run structure (word-spaced prose is one
  run, no vocabulary match), vocabulary (prose colon phrases are not table
  keys), parse (typed fields). Free-text residual: a vocabulary colon label, a
  wide-enough gap to split runs, a sole letters-only tail — zero instances in
  77 + filled dev (the sweep is the record); the confirm instructional docs
  are the falsification bed and move nothing.
- **The empty cell** (seattle BLANK, expect_absent): `Employee Name:` is
  followed by the colon barrier `Job Title:` with nothing between — zero runs,
  refused; the blank stays all-abstain (pinned end-to-end).
- **A real header row with a colon-terminated cell**: would weaken the amnesty's
  premise; measured, no column-header row in 77 + filled dev prints a
  colon-terminated cell (the amnesty additionally only ever engages on the
  recognised label's own line, never on the row above a table).
- **Over-reach into the row reader**: `column_value` is untouched; the license
  cannot pair a label with a value across baselines.
- **The two-candidate tie**: unchanged `len(runs) != 1` refusal, pinned by test.
- **`TO:` on the seattle forms** (`TO: ( Name & address of employer )`): gap
  61pt — already reachable today; the parse gate (date) refuses it today and
  keeps refusing it under the license. Byte-identical, in the sweep.
