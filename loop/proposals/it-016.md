# it-016 — T29 — the `<date> to <date>` pay-period range on a caption line — NO_SAFE_RULE (extraction) + successor T30

## 1. Target

`loop/backlog.json` T29 (this iteration files it). The pay period printed as a RANGE on
the line a pay-period label captions: `Pay Period: 1/7/XX to 1/13/XX`,
`… 8/21/2005 to 8/27/05`. The named corpus witness is **ca_dlse** (both stubs), where
`pay_period_start` and `pay_period_end` are abstained today. The stated stakes: the
deterministic path reads `gross_pay = 774.85` on ca_dlse_piecerate, but without a period
or frequency that gross never annualizes, so a piece-rate worker's income never completes.

## 2. The printed evidence (and the diagnosis it forces)

Two things the page prints are established by measurement, not assertion:

- **The label is present and is recognised.** `PAY PERIOD` is in `LABEL_MAP` (→
  `pay_period_start`); `Pay Period:` normalises (colon-folded) to the same key and is
  recovered by the synonym pass. Both ca_dlse stubs' period labels are recognised runs.
  So the abstention is **NOT a vocabulary miss** — answering the brief's first fork.
- **The value is a `<date> to <date>` span sitting on the CAPTION line** (the line below
  the label), not beside it. `core.extract._period_span_fields` already reads exactly this
  span shape — but only **side-by-side** (via `_side_by_side_run`, same line as the
  label). It fires on il_dol (`Pay Period: 8/21/2005 to 8/27/05`, one line) and reads
  `pay_period_start = 2005-08-21` today. ca_dlse puts the span on the next line, which no
  value branch (column / side-by-side / caption / columns / header-cell / span) splits, so
  the field takes no record and the terminal sweep prints the generic (here imprecise)
  "no label for this field was found on the page." Answering the second fork: **the label
  is present; the value SHAPE is unparsed in the caption position.**

The decisive third fact — **ca_dlse's years are the literal placeholder `XX`.** Truth
itself is `1/7/XX` / `1/13/XX` (confirm_truth). Both halves fail `_parse_date`; emitting
any ISO date would invent ≥ two digits (constraint 3, T6/MASKED_OUT). Abstaining on both
dates is the **correct** answer, not a defect. Reading the caption span here changes
nothing: the split would still abstain on both halves.

## 3. The license (class 1 — printed label + printed conjunction)

The rule the brief describes is genuinely licensed: a run `<date> <conj> <date>` under a
printed period label, with `conj` a printed word from a closed set, binds left→start,
right→end, each half read through the unchanged `_parse_date`, no year invented. This is
the same discipline as `_period_span_fields`, `total_band`, and the colon-line rule. The
license is not in question. Its **reachability on this corpus** is.

## 4. The rule that could be built — and why it is not built here

Two candidate extensions, each refused on measurement, not taste:

- **(a) Extend the span reader to the CAPTION line** (fire when the label's captioned
  value run is a `<date> to <date>` span). Licensed. But the falsification sweep (§6)
  finds the **only** caption-geometry span in 252 documents is ca_dlse, whose years are
  masked → the extended reader produces abstain/abstain, **byte-identical to today**. Zero
  fields flip; zero real-document positive evidence exists. Shipping it is a capability
  with no consumer and no witness — precisely the fitted, unfalsifiable change the loop
  exists to refuse ("a rule the corpus happens never to contradict").
- **(b) Widen the conjunction set to `through` / `thru` / dash.** The sweep finds **zero**
  occurrences of any conjunction other than `to` across all 252 documents. Adding them is
  a fitted constant with no printed witness — forbidden by the brief's own instruction
  ("never a fitted constant").

So the extraction half of T29 ships **no core change.** The side-by-side `to` range is
already read (il_dol); the caption case is masked (ca_dlse, correct abstain); nothing else
in the range family is reachable.

## 5. The hazards, named

**H1 — a `<date> to <date>` in a NON-period context bound as a pay period** (a
date-of-birth range, a coverage window, prose "from X to Y"). This is the rule's one way
to a WRONG value. *Refusal (structural):* the span reader only ever runs for a
`pay_period_start` label visit, so a non-period range is never offered to it. *Proof:* §6
finds **zero** non-period ranges anywhere — the hazard does not even materialise on the
corpus, so the structural refusal is never load-bearing here.

**H2 — inventing a year.** Any ISO date emitted for `1/7/XX` invents digits. *Refusal:*
both halves go through `_parse_date` unchanged; `XX` fails; the half abstains. Not built,
so not at risk — but recorded because it is the reason (a) yields nothing on ca_dlse.

## 6. Firing predicate

`loop/falsify/it-016.py`, swept in `loop/falsification/it-016.json`. Over 65 non-raster
manifest documents + 187 filled/scenario dev documents (252 total), it finds every
`<date> <conj> <date>` run, classifies each by conjunction, by whether both years are
resolvable (a real date could be emitted), and by whether it sits in a period context.
Result: **3 range-lines, all `to`, all period-context, 0 with resolvable years, 0
non-period (hazard) hits.** The three are ca_dlse_hourly (masked, glued `1/7/XXto`,
caption), ca_dlse_piecerate (masked, caption), il_dol (2-digit right year, side-by-side —
already read). Read-only, no model, no network.

## 7. Predicted flips (exact)

**None.** No code changes; the deterministic output is byte-identical on all 77. This is a
baseline-self-check iteration (like it-014): G7's target-mode "observed ∩ target ≠ ∅" is
**unsatisfiable by construction** here, because the only reachable target case (ca_dlse)
must remain an abstention. That unsatisfiability is itself the finding.

## 8. Modules touched (G6 allowlist) and the successor

Core/product code touched: **none.** Only `loop/` artefacts (proposal, falsify,
falsification, measurements, report, ledger, backlog).

**The real missing link is downstream, and it is filed as T30.** Even a perfectly-read
period does not complete the income: `logic/income.py` annualizes **only** from a stated
`pay_frequency` field (line 242) and its own commitment #1 states it will "refuse to guess
[a frequency] from pay dates"; it never reads `pay_period_start/end` (grep: zero
references in `logic/` or `pack/starter/`). The pack annualizes because pack stubs print
the frequency WORD (`biweekly`); real published stubs (il_dol, ca_dlse) print the RANGE
and no word, and `pay_frequency` is in ca_dlse's `expect_absent` — so emitting it as a
document field would be WRONG. Deriving frequency from the day-span (`1/7`→`1/13` = 7-day
= weekly, computable from month/day even when the year is `XX`) belongs in the annualization
layer as a derived quantity, against a standing design commitment — a policy/logic
successor, not an extraction rule. **T30.**
