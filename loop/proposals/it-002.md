# it-002 — T8 — a vocabulary label that is a printed header-row cell licenses its column

## 1. Target

Backlog **T8**, corpus `confirm`, doc `ca_dlse_paystub_hourly.pdf`. Fields:
`person_name` (truth `Johnson, Bob`), `hourly_rate` (truth `18.00`), both `abstain`.

## 2. The printed evidence (verbatim from `loop/worktmp/ca_dlse_paystub_hourly.words.json`)

Baseline 781.0 prints a row of four caption cells, 29 pt non-bold, split exactly at its
column gaps (`_split_runs` threshold 0.6 × 29 = 17.4 pt; intra-cell gaps 9–10 pt,
inter-cell gaps 78–172 pt):

    EMPLOYEE[228.0–397.4]  SOCIAL SECURITY NO.[569–910]  PAY RATE[988.5–1140.2]  PAY PERIOD[1239–1426]

Baseline 742.0, the single data row, 39.0 pt below:

    Johnson, Bob[227.4–414.4]  XXX-XX-6789[570–759]  18.00[989.0–1064.6]  regular[1087–1182]  1/7/XXto 1/13/XX[1239–1480]

`Johnson, Bob` shares x-extent with `EMPLOYEE` and no other cell; `18.00` with
`PAY RATE` and no other cell. The next row (`27.00 overtime`, baseline 703.0) is 78 pt
below the header — outside `HEADER_SEARCH_BAND` (60.0).

**Why nothing fires today — measured, and it corrects T8's root-cause note.** The label
path DOES recognise these cells: `_label_runs` at baseline 781.0 returns `EMPLOYEE`
(LABEL_MAP), `PAY RATE` (LABEL_SYNONYMS), `PAY PERIOD` — the vocabulary path is
typography-independent, so "not recognised at 29 pt non-bold" is the wrong mechanism.
What fails is every value rule downstream: `_resolve_value` looks 6–22 pt below the
label (`VALUE_Y_WINDOW`, fitted to the pack's 14.5–15.3 pt pitch) and the data row is
39.0 pt down; `_side_by_side_value` and `columns.column_value` read across the row,
where only other header cells sit; `columns.table_cell_value` for `hourly_rate` needs a
`REGULAR`-named row under a `RATE` header cell, and the only REGULAR row (baseline
589.0) sits under `EARNINGS|HOURS|AMOUNT|DEDUCTIONS|AMOUNT`, which prints no RATE cell;
the arithmetic path locates 18.00 by label adjacency but closes no identity (the rate is
not printed on the earnings rows) and correctly emits only an S2 proposal-abstention.

## 3. The license (exactly one of the five): **printed header row**

The page itself declares baseline 781.0 a row of column names: it is a row of ≥3 short,
digit-free caption cells, the exact structure `_header_row_words` already recognises —
today only to REFUSE those cells as values. A vocabulary label that is such a cell is a
column header by the page's own statement, and the page also states which column each
data run belongs to: the run's x-span shares extent with exactly one header cell. That
attribution is containment/uniqueness over printed cell spans — the same license
`columns._attribute_axis` already carries — not proximity. No new strings: the label
must already be in LABEL_MAP/LABEL_SYNONYMS. No new constants: `HEADER_SEARCH_BAND`
(60.0), `MIN_HEADER_ROW_CELLS` (3) and the caption tests are reused, not refitted.

Alternatives weighed and refused:
- *Scale `VALUE_Y_WINDOW` by page or font size*: replaces one fitted constant with
  another fitted constant, and relaxes geometry for every label on every page, not just
  printed header rows — the reach-the-next-form-row hazard. Refused.
- *Close `hourly_rate` by cross-table arithmetic* (18.00 × 40.00 = 720.00 and
  27.00 × 2.00 = 54.00 are both printed): a real license, but it reaches only the typed
  field and would leave `person_name` needing a second mechanism anyway. Refused as
  narrower, not as wrong.

## 4. The rule

New `columns.header_cell_value(lines, label_run, field_name, ...)`, called from
`_scan_page` immediately after the `columns.column_value` branch, under the same
abstention-only gate. It fires only when the label run **is** (word-for-word) one whole
cell of a line `_header_row_words` marks as a column-header row. Candidates: every run
on lines 0 < Δbaseline ≤ `HEADER_SEARCH_BAND` below the label, on the label's page,
excluding label runs ("a label is never a value"), excluding runs `_caption_refusal`
refuses, keeping only runs whose x-span shares extent with **exactly one** header cell
of that row — ours. For a typed field, a run that does not `parse_value` as the field's
type is not a candidate and does not trigger refusals (the `_is_numeric_run` precedent);
for a free-text field there is no such filter — every surviving run counts. **Exactly
one candidate or nothing.** The value goes through the unchanged `_build_value_field`
(caption refusal + type parse) and is capped at `certainty="low"` like every other
column rule. Flag: **`REALDOOR_HEADER_CELL`** — on by default; the branch is also
inside the existing `_columns_enabled()` gate, so `REALDOOR_COLUMNS=0` keeps its
promise that `core/columns.py` is never imported; either flag at `0` restores
bit-identical behaviour.

## 5. The hazard, named

**A data matrix read as an identity field.** The wrongest value this rule could
manufacture is a `person_name` from a table whose `Employee` header names a money
column — `osu_sample_earnings_statement.pdf` p2 prints
`Deduction Code | Description | Employee[389.0–425.1] | Employer` over five deduction
amounts, the document family that once shipped `person_name = "Employer"` at high
certainty. What refuses it, measured with coordinates: (a) **attribution** — the
decimal-aligned amounts (`88.02`[440.0–461.4], `1,885.62`[429.4–461.4], …) overhang the
cell span and share x-extent with zero header cells, so none is ever attributed; (b)
**the inherited caption refusal** — on `orangeusd_sample_paystub.pdf`, `Employee
ID`[41–74] IS attributed to the `Employee Name`[28.3–76.7] cell at Δ20.8 pt, and
`_caption_refusal` refuses it as a cell of another header row — the exact refusal built
after this family shipped wrong names; (c) **multiplicity** — a roster or matrix with
two candidate rows in band refuses on count (`up_024_pay_stub_table.pdf`'s
hours/rate/gross columns, two data rows each, measured to stay abstaining); (d) the
free-text shape test is deliberately NOT a candidate filter, so it can never eliminate
a rival — it can only refuse the sole survivor. A second, quieter hazard: the rule runs
before the page-level `table_cell_value` and could preempt its two-axis answer; swept
over all 77 documents it engages on utep/osu/orangeusd and yields zero candidates on
each, so nothing is preempted anywhere in the corpus.

## 6. Firing predicate

`loop/falsify/it-002.py` — read-only. `fires(doc)`: runs the full extractor twice
in-process (rule monkeypatched in at exactly the `_scan_page` call site vs. not), fires
iff the emitted field set differs, reporting every changed field with both readings.
`conflicts(fired)`: joins each changed field against that document's truth
(gold/manifest/truth-file shapes handled as in it-001); an emitted field in
`expect_absent` or differing from truth is a conflict. Sanity: must fire on the target.

- `core/test_columns.py` — new test class: (a) a header-row fixture at 29 pt non-bold
  with the data row 39 pt down reads `person_name` and `hourly_rate` at `low`; (b) two
  data rows under one cell abstain; (c) `REALDOOR_HEADER_CELL=0` abstains; (d) a
  caption-shaped run below is refused for free text; (e) a two-cell line is not a
  header row and never fires; (f) a non-parsing run under a typed cell is not a
  candidate; (g) a masked date under `PAY PERIOD` stays an abstention.
