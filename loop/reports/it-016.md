## Diagnosis (why ca_dlse abstains) — answering the brief's fork

Not a vocabulary miss. `PAY PERIOD` is in `LABEL_MAP`; `Pay Period:` normalises (colon
folded) to the same key via the synonym pass. Both ca_dlse stubs' period labels are
recognised runs (measured). The abstention is the **value SHAPE in the caption position**:
the `<date> to <date>` span sits on the line the label captions (below it), and
`core.extract._period_span_fields` — which already reads this exact span — fires
**side-by-side only** (`_side_by_side_run`, same line). It reads il_dol
(`Pay Period: 8/21/2005 to 8/27/05`, one line) → `pay_period_start = 2005-08-21` today, and
misses ca_dlse's below-label geometry, so the field takes no record and the terminal sweep
prints the generic (here imprecise) "no label found".

**The decisive third fact:** ca_dlse's years are the literal placeholder `XX` (truth
itself is `1/7/XX`). Both halves fail `_parse_date`; any ISO date invents ≥2 digits
(constraint 3, T6/MASKED_OUT). **Abstaining is the correct answer.** Reading the caption
span changes nothing here.

## The XX-year decision

Do not manufacture a year. Both ca_dlse halves abstain — as required. The frequency
SIGNAL is separable (a `1/7`→`1/13` day-count is 7 days = weekly regardless of year), but
that is not a `pay_period_start` and not a `pay_frequency` field (see downstream).

## Why no rule ships (both candidate extensions die on measurement)

- **Extend the span reader to the caption line** — licensed, but the only caption-range in
  252 docs is ca_dlse (masked → abstain/abstain, **byte-identical to today**): zero fields
  flip, zero real-document witness. A capability with no consumer and no test doc is the
  fitted, unfalsifiable rule the loop exists to refuse.
- **Widen the conjunction set to `through`/`thru`/dash** — **zero** occurrences of any
  conjunction but `to` across all 252 docs. A fitted constant with no printed witness;
  forbidden.

The side-by-side `to` range is already read (il_dol); the caption case is masked. Nothing
in the range family is reachable in extraction.

## The downstream, traced on a real (non-XX) document — il_dol

Reading the period does **not** complete the income, and the gap is not extraction.
`logic/income.py` never reads `pay_period_start/end` (grep: 0 references in `logic/` or
`pack/starter/`), annualizes only from a **stated `pay_frequency` field** (line 242), and
its commitment #1 states it will "refuse to guess [a frequency] from pay dates." Measured
end-to-end: a household with il_dol's `pay_period_start=2005-08-21`,
`pay_period_end=2005-08-27` and `gross_pay=440` all present →
`derive_wage_source` returns `annual=None`, trigger `pay_frequency_not_stated`
("it cannot be inferred"), **household total `None`.** The pack annualizes only because
pack stubs print the WORD `biweekly`; real published stubs print the RANGE and no word.
So: **extraction reads what it can, but income still does not complete because the
annualization layer requires a stated frequency and refuses to derive one from the span.**
Filed as **T30** (a policy/logic successor, not an extraction rule; and NOT emitting
`pay_frequency` as a field — it is in ca_dlse's `expect_absent`, so that would be WRONG).
