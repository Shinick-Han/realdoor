## The three design questions, settled on evidence
**Library: pypdf.** pdfminer (pdfplumber's engine) exposes annotations as raw
dicts with no field-inheritance resolver and cannot write; pypdf resolves `/V`
through the field tree and can rewrite the document, which the appearance-stream
approach requires. The *text geometry* still comes from pdfminer, because the
derived page is read by `read_words`.

**Appearance streams: every filled text widget has one, value baked in as a
`Tj`.** No double-read today — pdfplumber never descends into annotation
appearances (proven: `extract_text()` on the filled carrier contains neither
"Terrence" nor "1,283.15"). A *flattened* form would be a real double-read, and
`_drop_double_read` carries it.

**Labels: widget names are never consulted.** The carrier names a widget
`RATE OF PAY OR SALARY HOURLY DAILY OR PIECE RATE` and fills it `$17.10 hourly`;
binding by that name would light all 7 fields today. Refused — an internal name
is not printed on the page, varies per carrier with no vocabulary anyone
controls, and a *stale* name binds a real value to the wrong field with an exact
coordinate attached. So the yield is 1 field per carrier, not 7: the values are
all read and sit in the word stream, and the other six abstain because the
form's printed captions are outside the vocabulary. That is **T21**, now
reachable for the first time — the values are there, only the labels are not.

## Two findings that would have shipped a silent defect
* **`/Rect` is not the value's box.** `EMPLOYEES NAME` has a 210.7×21.5pt rect
  holding a string that sets to 71.6×12pt. Using the rect feeds every geometric
  rule a box three times too wide. Hence the appearance-stream construction.
* **`ocr_words.drop_overlapping` is the wrong guard here.** It drops on *any*
  intersection — right for tight OCR quads, wrong for typeset em boxes, which
  graze between adjacent lines. Applied here it silently deleted 5 of 17 values
  including both money fields (`$17.10` overlaps the caption above it by 1.4pt,
  7% of its area). Replaced by same-string-same-place ∪ ≥50% coverage; the
  measured worst real adjacency is 0.075, so the second leg is dormant.
* A third, caught in construction: without the AcroForm `/DR` merged under each
  appearance's `/Resources`, pdfminer resolves no font and returns **correct
  text at zero width**, silently. Pinned by test.

## The hazard I did not create and must not paper over
**`loop/baseline.json` was rebuilt at 04:16, mid-iteration, from a tree
containing my uncommitted change with `REALDOOR_FORM_FIELDS` defaulting ON.** It
now records `person_name = correct` for the filled wa_dshs carrier and all six
acroform scenario letters — fields that read `abstain` on master. My unaccepted
change has been laundered into the file that *defines* the accepted state, with
no accept decision, which is precisely what `field_state` exists to prevent. Its
`pytest` also records `passed: -1, exit_code: 2`, a broken collection, which
would make G4 trivially satisfiable. **This commit deliberately excludes
`loop/baseline.json` and `loop/tools/`.** I did not re-run the gate against the
rebuilt baseline: it would report a far better G7 that means nothing, because
the baseline was built from my own tree. The baseline must be rebuilt on a clean
master before any iteration trusts it again.
